# AGENTS.md

Guidance for coding agents working in `api-contract-app`.

## Repo Overview
- Bun workspace monorepo.
- `apps/web`: Vite + React 19 + TanStack Router + TanStack Query + Tailwind v4.
- `apps/api`: Hono + Bun + Drizzle ORM + SQLite.
- `packages/types`: shared TypeScript types used across web and api.

## Setup
Run from repo root:

```bash
bun install
cp .env.example .env
bun run --filter api db:migrate
```

## Build / Dev / Check Commands

### Root
- Start both apps: `bun run dev`
- Start web only: `bun run dev:web`
- Start api only: `bun run dev:api`
- Build all workspaces: `bun run build`
- Typecheck all workspaces: `bun run typecheck`

### Web
- Dev server: `bun run --filter web dev`
- Build: `bun run --filter web build`
- Preview build: `bun run --filter web preview`
- Typecheck: `bun run --filter web typecheck`
- Lint: `bun x eslint apps/web/src apps/web/vite.config.ts --ext .ts,.tsx`

### API
- Watch mode: `bun run --filter api dev`
- Run once: `bun run --filter api start`
- Typecheck: `bun run --filter api typecheck`
- Generate migrations: `bun run --filter api db:generate`
- Run migrations: `bun run --filter api db:migrate`
- Open Drizzle Studio: `bun run --filter api db:studio`

### Shared Types
- No dedicated scripts exist in `packages/types`.
- Validate shared type changes with `bun run typecheck`.

## TypeScript Rules From The Repo
- Web and api both use strict TypeScript.
- API enables `strict`, `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`, and `noImplicitOverride`.
- Web enables `strict`, `noUnusedLocals`, `noUnusedParameters`, `noFallthroughCasesInSwitch`, and `noUncheckedSideEffectImports`.
- Do not disable strictness to land a change.
- Avoid `any`; prefer inference, generics, discriminated unions, or explicit interfaces/types.
- Use `import type` for type-only imports.

## Import Conventions
- Follow the existing grouping pattern:
  1. external packages
  2. internal app imports
  3. type-only imports
- Remove unused imports.
- Web code prefers the `@/` alias for `apps/web/src`.
- Shared types should come from `@repo/types`.
- API code generally uses relative imports within `apps/api/src`.

## Formatting Conventions
- Match existing source style:
  - 2-space indentation
  - double quotes
  - no semicolons
  - trailing commas in multiline literals
- Keep code readable rather than overly dense.
- Prefer early returns over nested conditionals.
- Add comments only for non-obvious behavior or section markers already used nearby.

## Naming Conventions
- React components: PascalCase.
- Route modules export `Route`.
- Page components typically use names like `DashboardPage`, `ProjectIndexPage`, `LoginPage`.
- Hooks use `useX` naming.
- React Query key registries use `<feature>Keys`.
- Web API helpers use verb names like `getProject`, `createProject`, `updateProject`.
- Server route collections use names like `projectRoutes`, `authRoutes`.
- Constants use uppercase snake case only when truly constant.
- File names are usually kebab-case, except TanStack Router path segments.

## Frontend Conventions
- Prefer function components and hooks.
- Keep auth gating and redirects in TanStack Router `beforeLoad` where appropriate.
- Prefer feature-local hooks for server state instead of raw `fetch` inside components.
- Use TanStack Query for server state, invalidation, and optimistic updates.
- Reuse shared primitives from `apps/web/src/components/ui/` before creating new ones.
- Use `cn()` for Tailwind class merging.
- Preserve the existing visual language unless the task is explicitly a redesign.

## Backend Conventions
- Keep route handlers in `apps/api/src/routes`.
- Validate request payloads with `zod` and `zValidator`.
- Use `requireAuth` middleware instead of duplicating auth checks in each handler.
- Keep DB access in Drizzle query-builder form.
- Prefer small helpers in `apps/api/src/lib` for reusable behavior.
- Use `generateId()` for entity IDs.

## Error Handling Patterns
- API handlers return JSON error objects like `{ error: "Message" }` with the right HTTP status.
- Web API calls go through `apiFetch()` in `apps/web/src/lib/fetch.ts`.
- `apiFetch()` throws readable `Error` messages for non-OK responses.
- Catch errors only where the caller can recover or show UI feedback.
- Silent fallback is acceptable only when intentional, such as auth probing in `getMe()`.
- Prefer visible toast or inline error states over console-only failures.

## Data / Type Boundaries
- DB schema and row types live in `apps/api/src/db/schema.ts`.
- Shared transport/domain interfaces live in `packages/types/src`.
- Keep API payload shapes aligned with `@repo/types` when practical.
- When changing request or response shapes, update both server code and shared/web consumers.

## Safe Editing Habits
- Prefer small, local edits over broad rewrites.
- Read adjacent files before changing conventions.
- Do not revert unrelated user work.
- Do not manually change generated route tree output.
- Prefer changing source routes/components and letting tooling regenerate artifacts.

## Suggested Agent Workflow
1. Read the relevant feature module, route, and shared types.
2. Check for existing utilities before adding new helpers.
3. Make the smallest coherent change.
4. Run the narrowest useful verification command.
5. For cross-app changes, finish with `bun run typecheck`.
